\section*{Slide 3 — The Security Gap in Smart Grid Hosts \hfill \normalfont\small $\approx$1:30}
\addcontentsline{toc}{section}{Slide 3 — The Security Gap in Smart Grid Hosts}
\markboth{Slide 3 — Motivation}{Slide 3 — Motivation}

Modern smart grids are no longer air-gapped. They integrate IP-connected Windows workstations running SCADA software, human-machine interfaces, remote terminal units, and historian servers---all within the same network fabric as corporate IT. This convergence has been exploited in some of the most consequential cyber attacks of the last fifteen years. Stuxnet in 2010 targeted Windows hosts embedded in nuclear centrifuge control loops. The 2015 and 2016 Ukrainian grid attacks combined spear phishing with lateral movement through the IT/OT boundary to physically disconnect power to hundreds of thousands of homes. Colonial Pipeline in 2021 showed that IT-side ransomware could force an entire pipeline operator to manually shut down OT systems out of caution. \textit{[click]}

The regulatory response has been thorough. NERC CIP-007 mandates event logging on all applicable cyber assets in the bulk electric system. IEC 62351 specifies audit trail requirements for power system communications. Utilities are legally required to collect Windows Event Logs from their SCADA hosts. \textit{[click]}

The problem is that these logs are almost never used for automated intrusion detection. They are collected, stored for compliance, and ignored from a security monitoring perspective. The figure on the right shows the IT/OT architecture we target---Purdue model Levels one through three. Our goal is to close that gap using machine learning on the event logs that are already being generated. \textit{[click to next slide]}

\section*{Slide 4 — Prior Work and Research Gap \hfill \normalfont\small $\approx$1:30}
\addcontentsline{toc}{section}{Slide 4 — Prior Work and Research Gap}
\markboth{Slide 4 — Background}{Slide 4 — Background}

Three research streams are directly relevant, and each one leaves a gap we fill. The first is log anomaly detection: tools like Drain, DeepLog, and LogAnomaly have achieved impressive results on distributed system logs---HDFS, Linux syslog, OpenStack. But none of them were designed for Windows Security Event IDs, which carry structured semantics about authentication, process creation, and object access that generic log parsers do not exploit. They also do not model ICS-specific threat patterns.

The second stream is ICS intrusion detection. Carcano and colleagues used critical-state analysis on SCADA command sequences; Goldenberg and Wool built finite automata over Modbus/TCP traffic; the SWaT dataset from Singapore provides the community's main ICS benchmark. But all of these work at the network or physical process level---none of them look at Windows host audit logs.

The third stream is Windows malware detection on IT endpoints---recent work from 2024 and 2025 has used Sysmon logs, event ID sequences, and graph-based provenance analysis. The gap here is the operational context: OT systems run legacy Windows versions, have different software baselines, and generate fundamentally different event patterns than corporate IT hosts. \textit{[pause]}

Mitchell and Chen confirmed in a 2014 ACM Computing Surveys paper that host-level intrusion detection is rare in the cyber-physical systems literature. That finding still holds. Our work is, to our knowledge, the first framework specifically targeting Windows Event Logs at ICS Purdue Levels one through three in a smart grid environment. \textit{[click to next slide]}

\section*{Slide 5 — Threat Model \hfill \normalfont\small $\approx$1:00}
\addcontentsline{toc}{section}{Slide 5 — Threat Model}
\markboth{Slide 5 — Threat Model}{Slide 5 — Threat Model}

Our adversary model is grounded in MITRE ATT\&CK for ICS. We assume the attacker has already established a foothold on a Level~3 IT host---through spear phishing or a supply-chain compromise---and possesses valid or cracked domain credentials. They have network access across the IT/OT boundary through a misconfigured DMZ or a legitimate remote-access tunnel that has been hijacked. Their goal is to persist on the OT network, escalate privileges, and ultimately execute a disruptive payload---either rogue commands to a PLC or denial of operator visibility.

We model four specific attack patterns that produce distinct Windows Event Log signatures. Lateral movement through credential brute-forcing generates repeated Event~4625 logon failures followed by a 4624 success from an unexpected source IP. Persistence through malicious service installation generates Event~7045 on a SCADA host followed by 7036 state change. Privilege escalation generates a 4672 special-privileges assignment immediately preceding a 4688 process creation with an anomalous binary name---mimikatz or psexec. And reconnaissance generates a burst of 8 to 20 Event~4663 object-access records within 30 seconds on HMI or RTU hosts. Each of these becomes a learnable pattern in our feature space. \textit{[click to next slide]}

\section*{Slide 6 — Synthetic Dataset: Why and How \hfill \normalfont\small $\approx$1:30}
\addcontentsline{toc}{section}{Slide 6 — Synthetic Dataset: Why and How}
\markboth{Slide 6 — Dataset}{Slide 6 — Dataset}

Before discussing the generator, I want to address the question you may already be thinking: why synthetic data? The answer is simple---no publicly available Windows Event Log dataset exists for smart grid SCADA or RTU hosts. Conti and colleagues confirmed this in a comprehensive 2021 survey of ICS security testbeds and datasets. Nafees and colleagues attribute the gap to liability constraints and the unwillingness of energy operators to disclose network topology through released logs. This is not a problem unique to our work. Goh and colleagues built the entire SWaT Secure Water Treatment testbed precisely because no real ICS attack data was available, and that dataset is now the standard benchmark in the field. Chen and colleagues in 2022 synthesised APT-style Windows audit sequences for the same reason. We follow the same precedent.

Our generator produces 6{,}000 labelled sessions: 5{,}000 benign and 1{,}000 malicious, for a total of 89{,}288 individual events at a 5-to-1 class ratio. Benign sessions simulate realistic operator workflows---logon, interleaved object-access and process-creation events representative of SCADA client use, and a terminal logoff, with inter-event gaps drawn from a distribution modelling normal operator think-time. Importantly, we also include evasive attack variants: rate-limited lateral movement that spaces credential failures 60 to 300 seconds apart, and slow reconnaissance that extends burst intervals. These variants prevent the model from relying on a single trivially separable feature and make the evaluation more realistic. The pipeline figure on the right shows the full flow from generation through evaluation. \textit{[click to next slide]}

\section*{Slide 7 — Session-Level Feature Engineering \hfill \normalfont\small $\approx$1:30}
\addcontentsline{toc}{section}{Slide 7 — Session-Level Feature Engineering}
\markboth{Slide 7 — Features}{Slide 7 — Features}

The core of our approach is the session-level feature engineering pipeline. Rather than treating the event log as an unbounded stream and applying sequence modelling, we segment events into sessions using a 30-minute inactivity timeout---standard practice in network session analysis---and extract a compact 24-dimensional feature vector from each session.

The table on the slide shows the five feature groups. The first group computes occurrence counts for each of the 13 Windows Event IDs in our schema, plus a total event count---14 dimensions. The second group computes time-delta statistics: the mean, standard deviation, minimum, and maximum of the inter-event gap in seconds within each session---4 dimensions. This is where reconnaissance and lateral-movement attacks are most visible: a burst of 20 object-access events in 30 seconds produces dramatically different timing statistics than normal operator sessions separated by minutes of think-time. The third group captures the number of unique source IPs, usernames, and hostnames within a session---3 dimensions---serving as a lateral-spread indicator. The fourth group flags event IDs that appear in fewer than 5 percent of benign training sessions as binary indicators. The fifth group computes Shannon entropy over the event-ID distribution within the session.

Together these 24 features require no scaling, handle mixed types natively in a tree ensemble, and keep inference latency in the millisecond range---well within the requirements of SIEM integration. \textit{[click to next slide]}

\section*{Slide 8 — Classification Results \hfill \normalfont\small $\approx$2:00}
\addcontentsline{toc}{section}{Slide 8 — Classification Results}
\markboth{Slide 8 — Results}{Slide 8 — Results}

We compare Random Forest against five baselines, ranging from an unsupervised Isolation Forest to XGBoost. Let me walk through the table. The Isolation Forest, which receives no attack labels during training, achieves macro F1 of 0.601---this is the no-label lower bound, representing what an anomaly detector could achieve if labelled attack sessions were unavailable, as in a zero-day scenario. Logistic Regression reaches 0.942, and SVM with an RBF kernel reaches 0.844---both confirming that the features are not perfectly linearly separable, which is consistent with the evasive attack variants we designed. Decision Tree achieves 0.984, showing that simple tree rules capture most of the signal. XGBoost is a very strong baseline at 0.998---it nearly matches Random Forest. The Random Forest achieves macro F1 of 0.999 and ROC-AUC of 1.000.

The model was selected because it handles mixed feature types without normalisation, the class\_weight=balanced setting compensates for the 5-to-1 imbalance without oversampling, feature importances provide interpretability, and sub-millisecond inference per session makes deployment feasible. Hyperparameters were tuned via RandomizedSearchCV with 20 configurations and 5-fold stratified cross-validation maximising macro F1.

Critically, to rule out a lucky data split, we repeated the full pipeline---generate, preprocess, featurize, train, evaluate---across five independent random seeds. The mean F1 is 0.999 with standard deviation 0.001. All numbers in this table were computed from the actual pipeline; no values were manually entered into the paper. \textit{[click to next slide]}

\section*{Slide 10 — Ablation Study \hfill \normalfont\small $\approx$1:30}
\addcontentsline{toc}{section}{Slide 10 — Ablation Study}
\markboth{Slide 10 — Ablation}{Slide 10 — Ablation}

The ablation study removes one feature group at a time, retrains with identical hyperparameters, and re-evaluates macro F1 on the same held-out test set. All numbers are computed, not estimated. The full model achieves F1 of 1.000.

Removing Group~1---event frequency counts---causes the largest drop: F1 falls to 0.944, a delta of negative 0.056. This is the most critical group. Removing Group~2---time-delta statistics---causes the second-largest drop to 0.972. Removing Groups~3 through~5---cardinality, rare-event flags, and sequence entropy---shows zero marginal impact.

I want to be clear about what zero delta means here: it does not mean entropy is uninformative. Tested in isolation, entropy achieves F1 of 0.77---that is real signal. The zero marginal delta means that given event counts and timing, entropy adds nothing further because those two groups already fully separate benign from malicious sessions in this dataset.

The practical implication is important for deployment. If you are deploying to a Windows system where timestamp resolution is coarse or unavailable---some legacy OT systems log events with one-second or one-minute granularity---you should prioritise event counts. On noisier real-world logs where counts and timing are less clean, we expect Groups~3 through~5 to recover importance. \textit{[click to next slide]}

\section*{Slide 11 — Conclusion and Future Work \hfill \normalfont\small $\approx$1:00}
\addcontentsline{toc}{section}{Slide 11 — Conclusion and Future Work}
\markboth{Slide 11 — Conclusion}{Slide 11 — Conclusion}

To summarise the contributions. We have presented the first open-source framework for Windows Event Log-based malware detection specifically designed for smart grid industrial control systems at Purdue Levels one through three. The framework includes a synthetic event log generator covering four MITRE ATT\&CK ICS attack categories with evasive variants, a 24-dimensional session-level feature engineering pipeline, and a class-imbalance-robust Random Forest classifier. The framework achieves macro F1 of 0.999 with standard deviation 0.001 across five independent seeds and ROC-AUC of 1.0. Event counts are the most discriminative feature group; removing them drops F1 by 0.056. Inference runs in under half a second per 900-session batch on a single CPU core, confirming operational feasibility.

The open-source nature of the framework directly addresses the data availability barrier that has limited ICS security research. Future directions include evaluation on real ICS event logs from controlled red-team exercises, extension to multi-class MITRE ATT\&CK technique attribution, online learning for concept drift, and adversarial robustness evaluation. \textit{[click to references, then Q\&A]}
